## 1. 프로젝트 개요

### 1.1. 배경 및 문제 정의

QA 테스트 자동화는 소프트웨어 품질 확보의 핵심이지만, 현장에서는 다음과 같은 진입 장벽이 존재한다.

| # | 문제 | 영향 |
|:-:|:---|:---|
| 1 | Selenium 스크립트 작성에 **Python + XPath/CSS** 등 개발 역량 필요 | 비개발자 QA 담당자가 자동화에 접근 불가 |
| 2 | 수동으로 셀렉터를 찾고 코드를 짜는 데 **시간 소모 과다** | 자동화 도입 ROI 저하 |
| 3 | 웹 요소 변경 시 테스트 깨짐, **유지보수 비용 높음** | 한 번 만든 테스트를 버리게 되는 악순환 |
| 4 | 반복적 데이터 테스트(DDT)에 별도 프레임워크 필요 | 학습 곡선 추가 부담 |

### 1.2. 솔루션: No-Code 테스트 자동화 플랫폼

**"코드를 한 줄도 작성하지 않고"** 브라우저에서 클릭/입력만으로 테스트 시나리오를 만들고, 이를 **완전한 Pytest + POM 구조의 Python 프로젝트**로 내보내는 데스크톱 도구.

```
사용자 클릭/드래그  →  자동 로케이터 분석  →  스텝 리스트 구성  →  Pytest 코드 생성  →  실행 & 리포트
```

### 1.3. 해결 매핑

| 문제 | AutoTestBuilder 해결 방식 |
|:---|:---|
| 셀렉터 모름 | 브라우저 요소 클릭 → **최적 로케이터 자동 분석** (data-testid > id > aria-label > CSS > XPath) |
| 코드 작성 불가 | GUI에서 스텝 추가 → **Pytest 코드 자동 생성** |
| 요소 변경 시 깨짐 | **Self-Healing** + 다중 Fallback 로케이터 |
| 데이터 반복 테스트 | 엑셀 파일 연동 → **DDT 자동화** |
| 리포트 필요 | **Allure + HTML 리포트** 동시 지원 |

## 2. 타깃 사용자 및 가치 제안

### 2.1. 주요 사용자 페르소나

| 페르소나 | 역할 | Pain Point | AutoTestBuilder 가치 |
|:---|:---|:---|:---|
| **QA 엔지니어 (비개발)** | 수동 테스트 담당 | 코드 작성 불가, 반복 업무 과다 | No-Code로 테스트 생성, DDT로 반복 제거 |

### 3.2. Playwright와의 심층 비교 및 격차 분석

Playwright는 현재 테스트 자동화 도구 중 가장 현대적이고 강력한 도구로, AutoTestBuilder와 비교 시 기술적 차이를 명확히 인식해야 한다.

#### 엔진 아키텍처 차이

| 항목 | AutoTestBuilder (Selenium) | **Playwright** |
|:---|:---|:---|
| 통신 프로토콜 | HTTP 기반 WebDriver 프로토콜 (2004~) | **CDP/DevTools 프로토콜** (2020~) |
| 자동 대기 | `WebDriverWait` 수동 구현 | 모든 액션에 **Auto-Wait 내장** |
| 네트워크 인터셉트 | ❌ 불가 | ✅ `route()` API |
| 다중 탭/팝업 | `window_handles` 수동 전환 | ✅ 네이티브 멀티 컨텍스트 |
| iframe 처리 | `switch_to.frame()` 수동 전환 | ✅ `frame_locator()` 자동 |
| 파일 업로드/다운로드 | 복잡한 워크어라운드 | ✅ `set_input_files()` |
| 실행 속도 | 상대적 느림 | **2~3배 빠름** |
| 브라우저 설치 | `webdriver-manager` 의존 | `playwright install` 한 줄 |

#### Playwright의 UX 장점 — 우리가 배워야 할 것

| Playwright 기능 | 현재 AutoTestBuilder | 개선 방향 |
|:---|:---|:---|
| **실시간 코드 생성** (Inspector) | 스텝 테이블에만 표시, 코드는 실행 시 생성 | → **코드 미리보기 탭** 추가 |
| **호버 스캔** | F2 키 + 창 전환 필요 | → **호버 하이라이트 모드** 도입 |
| **Step-by-Step 디버깅** | 전체 실행만 가능 | → **한 스텝씩 실행** 모드 추가 |
| **Trace Viewer (시간 여행)** | 실패 시 스크린샷만 | → **스텝별 스크린샷 타임라인** |
| **다국어 코드 생성** | Python 전용 | → JavaScript/TypeScript 지원 검토 |

#### AutoTestBuilder의 고유 강점 — 이것이 우리의 차별점

| 강점 | 설명 |
|:---|:---|
| **No-Code GUI** | Playwright는 결국 **코드를 작성해야** 한다. 우리 도구는 코드 없이 테스트 완성 가능. |
| **POM 프로젝트 내보내기** | BasePage → PageObject → TestScript 3계층 구조를 자동 생성. Playwright에서는 직접 구성해야 함. |
| **엑셀 DDT GUI** | GUI에서 직접 엑셀 데이터를 편집하고 변수 바인딩. Playwright는 코드로만 가능. |
| **Self-Healing 로케이터** | 요소 변경 시 Fallback 자동 시도. Playwright는 단일 로케이터 실패 즉시 오류. |
| **동적 클래스 자동 필터링** | CSS-in-JS 해시 클래스를 자동 제외. |

> **결론:** Playwright는 **개발자용 도구**이고, AutoTestBuilder는 **비개발자용 No-Code 도구**이다. 경쟁 관계가 아니라 **보완 관계**로 접근해야 하며, 장기적으로 **Playwright 엔진을 백엔드로 채택**하여 양쪽의 장점을 결합하는 전략이 최선이다.

## 5. 사용 가이드

### 5.1. 설치

```bash
# Python 3.8 이상 필요
pip install -r requirements.txt
```

### 5.2. 실행 방법

#### GUI 모드 (기본)
```bash
python main.py
```

#### CLI 모드 (CI/CD용)
```bash
# 기본 실행
python run_tests.py --test-file first.json

# 브라우저 지정 + 헤드리스
python run_tests.py --browser firefox --headless --test-file first.json

# DDT 데이터 지정
python run_tests.py --test-file first.json --data-path data.xlsx

# 커스텀 설정 사용
python run_tests.py --config my_config.yaml --test-file first.json
```

### 5.3. GUI 워크플로우

```
 1️⃣  브라우저 열기 → URL 입력 + 브라우저 선택 → [브라우저 열기] 클릭
       ↓
 2️⃣  테스트 스텝 작성 → 웹 요소 클릭/드래그 → F2 또는 [요소 스캔]
       ↓
 3️⃣  데이터 바인딩 (선택) → 엑셀 로드 → Value에 {변수명} 입력
       ↓
 4️⃣  실행 또는 내보내기 → [▶ 실행] 또는 [📦 POM 내보내기]
```

### 5.4. 설정 관리

설정 우선순위: **환경변수(`QA_ATB_*`)** > **config.yaml** > **기본값**

```yaml
browsers:
  default: chrome       # chrome | firefox | edge | safari
  headless: false       # CI/CD에서는 true 권장
test:
  parallel_workers: 1   # 병렬 워커 수
  retry_count: 1        # 실패 시 재시도
  timeout: 30           # 대기 타임아웃(초)
report:
  type: allure          # allure | html
  screenshot_on_failure: true
```

## 7. 현재 이슈 및 기술 부채

### 7.1. 코드 품질 이슈

| 영역 | 이슈 | 영향 | 우선순위 |
|:---|:---|:---|:---:|
| `qt_app.py` | 729줄 단일 파일에 3개 탭 로직 혼재 | 유지보수 난이도 증가 | 🟡 중 |
| `runner.py` | `subprocess` 직접 호출 (pytest API 미사용) | 에러 핸들링 취약 | 🟡 중 |
| `browser_config.py` | 문자열 결합 방식 코드 템플릿 | 새 브라우저 추가 시 복잡 | 🟢 낮음 |
| 전반 | 프로젝트 자체 유닛 테스트 부재 | 리팩토링 시 회귀 위험 | 🔴 높음 |
| 전반 | 타입 힌팅 불완전 | IDE 지원 약화, 버그 유입 가능 | 🟡 중 |

### 7.2. 기능 격차 (vs Playwright)

| 격차 영역 | 현재 상태 | 개선 방향 | 우선순위 |
|:---|:---|:---|:---:|
| 실시간 코드 미리보기 | 없음 | 코드 프리뷰 탭 추가 | 🔴 높음 |
| Step-by-Step 디버깅 | 전체 실행만 가능 | 한 스텝씩 실행 모드 | 🔴 높음 |
| Trace/Timeline | 실패 시 스크린샷만 | 스텝별 스크린샷 타임라인 | 🔴 높음 |
| 로그 시각화 | 단색 텍스트 | 성공/실패/경고 색상 구분 | 🟡 중 |
| 호버 스캔 | F2 수동 스캔 | 마우스 호버 실시간 하이라이트 | 🟡 중 |
| 네트워크 모킹 | 불가 | Selenium Wire 또는 엔진 전환 | 🟡 중 |
| 비주얼 리그레션 | 없음 | 스크린샷 픽셀 비교 | 🟡 중 |
